# v0.2.0 LAN Distribution — Architecture Design

**Date:** 2026-07-21
**Status:** Approved — ready for implementation plan
**Scope:** Repackage MetalFlow ERP as a double-click-installable desktop app for Windows, where one PC acts as server (DB + Next.js + UDP discovery) and 2-5 client PCs connect over LAN via Tauri 2.x shells.

## Problem

MetalFlow ERP runs today as a hosted web app (Postgres on remote server). The user wants to deploy it on-premise at a small workshop (2-5 PCs in one LAN) with:

1. **Installable with double-click** on Windows — no terminal, no Node.js, no manual Postgres setup.
2. **One PC acts as server**, the rest connect to it over LAN as clients.
3. **Local-first** — DB and app live on the server PC's filesystem, accessible from LAN.
4. **Cross-version safe** — clients detect version mismatch before opening the webview.

## Why pivot

Previous design (2026-07-05 spec) used Tauri-bundled-server: Tauri shell spawned Node child process. After 2+ hours of complex Rust lifecycle work (`tauri::generate_context!()` macro can only be invoked once per crate, requiring per-role binaries; ~2.4GB per worktree of Rust target/ artifacts), the user requested:

> "podriamos hacer una solucion mejor con un server node js y los clientes con tauri o electron"

This design **eliminates Tauri on the server side entirely**. The server is a pure Node PKG bundle; Tauri 2.x is used only for the thin client shells.

## Goal

Ship two products:

1. **`metalu-server.exe`** (~80MB, PKG bundle with Node 20 embedded + Next.js standalone + PGlite). Doble-click arranca, listens on `0.0.0.0:3000`, broadcasts UDP discovery on `:3001`, owns DB file in `%APPDATA%/metalu/db/`.
2. **`metalu-client.exe`** (~8MB, Tauri 2.x shell). On launch, listens for one server UDP broadcast, opens WebView2 window pointing at `http://<server-ip>:3000`, caches URL for 30 days.

All existing app code (modules, services, APIs, validation, auth) continues to work without modification. PGlite speaks real Postgres SQL, so Prisma schema is **unchanged**.

## Non-Goals (YAGNI — explicit out-of-scope for v1)

- HTTPS / TLS on LAN (LAN assumed trusted).
- Auto-update (Fran distributes new `.exe` manually via Dropbox/USB).
- Hot failover (if server PC dies, replace manually with backup).
- Scheduled/automated backups (button-triggered only).
- Code signing + Apple notarization (deferred until outside-the-taller distribution).
- Multi-write Postgres / multi-node concurrency (PGlite single-writer — fine for 2-5 PCs).
- Multi-platform: **Windows x64 only** (Mac/Linux deferred).
- Hot-reload during dev (manual restart).

## Components

### Server (Node PKG bundle)

```
dist-server/metalu-server.exe             ← PKG output, single binary

scripts/
├── build-server-bundle.sh                ← pkg build (Windows x64 only v1)
└── dev-server.ts                         ← dev mode sin bundle

src/
├── server/
│   ├── entry.ts                          ← PKG entrypoint
│   ├── pglite-bootstrap.ts               ← ensure data dir + run migrations
│   └── shutdown.ts                       ← cleanup handlers (SIGINT/SIGTERM)
└── lib/
    └── discovery/
        ├── announce.ts                   ← UDP broadcast cada 5s (NEW)
        ├── payload.ts                    ← JSON schema: {ip, port, hostname, version}
        └── __tests__/announce.test.ts    ← vitest unit tests

src/app/api/
├── health/route.ts                       ← AC:17, ✅ cherry-picked
├── version/route.ts                      ← AC:12, ✅ cherry-picked
├── server-info/route.ts                  ← NEW: IP LAN + puerto + status
└── backup/
    ├── route.ts                          ← POST export, GET list
    └── [filename]/route.ts               ← GET download, DELETE remove
    └── restore/route.ts                  ← POST upload + import

src/app/admin/
├── layout.tsx                            ← layout admin
├── page.tsx                              ← dashboard admin (links)
├── server-info/page.tsx                  ← NEW
└── backups/page.tsx                      ← NEW: UI backup/restore

src/lib/backup/
├── export.ts                             ← pglite.export() → .pglitebackup
├── restore.ts                            ← read .pglitebackup → pglite.import()
└── __tests__/
```

**Key modules:**

- **`src/lib/discovery/announce.ts`** — `dgram.createSocket('udp4')`, `setInterval(broadcast, 5000)`. Reads local IP via `os.networkInterfaces()` (prefers IPv4 non-internal). Payload JSON validated against schema.
- **`src/server/entry.ts`** — PKG entrypoint: `require('next')()`, arranca en :3000, monta announcer, registra SIGINT handler.
- **`src/lib/backup/export.ts`** — `pglite.export()` returns Uint8Array, writes to `%APPDATA%/metalu/backups/metalu-YYYY-MM-DD.pglitebackup`.

### Client (Tauri 2.x shell)

```
src-tauri-client/                         ← SEPARADO del server (un solo binario)
├── Cargo.toml                            ← Tauri + tokio + serde only
├── tauri.conf.json                       ← window config (URL del server)
├── build.rs
├── icons/                                ← mismo logo.svg tamaño cliente
└── src/
    ├── main.rs                           ← entrypoint Tauri
    ├── discovery.rs                      ← UDP listener Rust (NEW)
    ├── server_url.rs                     ← cache a %APPDATA%/metalu-client/
    └── commands.rs                       ← Tauri commands (navigate)
```

**Por qué `src-tauri-client/` separado (no `src-tauri/`):**
- Server ya no es Tauri → no hay dual build
- Un único binario cliente, una sola config
- CI matrix más simple (1 job Win x64 vs 4 del spec viejo)

**Flujo del cliente (`src-tauri-client/src/main.rs`):**
```rust
fn main() {
    let cached = server_url::read_cache();
    let url = match cached {
        Some(u) if u.expires_at > now() => u.url,
        _ => discovery::wait_for_server(30_000),
    };
    server_url::write_cache(&url);
    tauri::Builder::default()
        .setup(|app| {
            WebviewWindowBuilder::new(app, "main", url)
                .title("Metalu Cliente")
                .inner_size(1280.0, 800.0)
                .build()?;
            Ok(())
        })
        .run(tauri::generate_context!())?;
}
```

## Data flow

### Server boot
```
[0ms]    PKG binary arranca Node embebido
[100ms]  src/server/entry.ts ejecuta require('next')
[500ms]  Next.js compila rutas on-demand
[1s]     src/lib/prisma/pglite.ts abre %APPDATA%/metalu/db/
[1.5s]   src/lib/prisma/migrations.ts corre migrations
[2s]     Next.js escucha en 0.0.0.0:3000
[2s]     src/lib/discovery/announce.ts crea UDP socket :3001
[2.5s]   Primer broadcast: {ip, port, hostname, version}
[7.5s+]  broadcasts cada 5s
```

### Client first run (sin cache)
```
[0ms]    Tauri main arranca
[50ms]   discovery::wait_for_server(30_000) abre UDP socket :3001
[~50-5000ms] Primer broadcast recibido
[~100ms] Valida JSON payload + GET /api/health (200 OK)
[~200ms] Escribe cache: %APPDATA%/metalu-client/server-url.txt
[~300ms] WebviewWindowBuilder abre URL → UI renderiza
```

### Client subsequent run (cache hit)
```
[0ms]    Tauri main arranca
[20ms]   server_url::read_cache() → URL válida
[50ms]   WebviewWindowBuilder abre URL
[300ms]  UI lista, sin discovery
```

### Admin backup
```
[click "Hacer backup ahora"]
  → POST /api/backup
  → src/lib/backup/export.ts
  → pglite.export() → fs.writeFileSync(.pglitebackup)
  → { ok, filename, sizeBytes }
```

### Admin restore
```
[click "Restaurar desde archivo" + file picker]
  → POST /api/backup/restore (multipart)
  → src/lib/backup/restore.ts
    1. Acquire restore.lock
    2. Stop accepting writes (in-memory flag)
    3. wait 1s (drain in-flight)
    4. fs.readFileSync + pglite.import()
    5. Release lock + re-enable writes
  → { ok, restoredFrom }
```

## Error handling

### Server-side

| Error | Manejo | UX |
|---|---|---|
| Puerto 3000 ocupado | PKG prueba 3001-3010, usa primero libre. Anuncia puerto real. | Transparente |
| PGlite write conflict (>5s timeout) | Service mapea a HTTP 503 + `Retry-After: 1` | Toast "Reintentando..." con backoff |
| PGlite corruption on startup | `pglite.open()` throws → log error + sugerir restore | Banner en /admin |
| Migration failure | Server sigue arriba, loggea error. Backup panel: "DB inconsistente" | Banner persistente |
| Disk full (no se puede escribir backup) | API devuelve 500 | Toast: "Disco lleno" |

### Client-side

| Error | Manejo | UX |
|---|---|---|
| Discovery timeout (>30s sin broadcast) | Tauri main muestra modal | Modal: "No se encontró servidor. Verificá que esté encendido y en la misma red WiFi." + botón "Reintentar" |
| Cache corrupta | Se descarta, vuelve a discovery | Transparente |
| HTTP 500 sostenido | Webview muestra error de Next.js con retry | Default Next.js error UI |
| HTTP 401 (token expirado) | next-auth redirige a /login | Transparente |

### Network

| Escenario | Manejo |
|---|---|
| LAN partition | Cliente no recibe UDP → discovery timeout → modal de error |
| Cliente pierde WiFi 10s | Webview fetch falla → toast "Conectando..." + retry cada 5s |
| DHCP renueva IP server | Cliente detecta por failure de fetch. Re-discovery al reiniciar app. |

### Data integrity

| Escenario | Manejo |
|---|---|
| Backup file corrupto | `pglite.import()` throws → 500. Backup NO se borra. |
| Restore parcial | PGlite rollback automático. UI: "Restore falló, DB sin cambios" |
| Dos restores simultáneos | Lock file `%APPDATA%/metalu/restore.lock`. Segundo: 409 Conflict |

- `announce.ts` — mock `dgram`, verifica broadcast cada 5s con payload correcto
- `payload.ts` — JSON schema validation, version compat check
- `backup/export.ts` + `restore.ts` — mock PGlite + fs, verifica cleanup en error
- `/api/server-info` — mock `os.networkInterfaces()`, verifica IP LAN

- `/api/backup/restore` POST — lock file + cleanup en error
- `src-tauri-client/src/discovery.rs` — mock UDP socket
- `src-tauri-client/src/server_url.rs` — cache read/write/expiración

### Integration tests (Vitest, PGlite real)
- `tests/discovery-e2e.test.ts` — 2 announcers + 2 listeners, verifica broadcasts cruzados
- `tests/backup-roundtrip.test.ts` — export → drop → import → datos idénticos
- `tests/api-health-server-info.test.ts` — coexistencia de endpoints

### Build verification (CI, GitHub Actions Windows x64)
- Job 1: `npm test` (unit + integration)
- Job 2: `pnpm pkg .` → assert `dist-server/metalu-server.exe` (60-120MB)
- Job 3: `cargo build --release` → assert `src-tauri-client/target/release/metalu-client.exe` (5-12MB)
- Job 4: Smoke — arranca .exe en background, espera 5s, GET `localhost:3000/api/health` → 200, mata proceso

### Manual smoke test (checklist, hardware real)
1. Instalar `metalu-server.exe` → ventana cmd parpadea → `localhost:3000` reachable
2. Login admin/admin, ver dashboard
3. Instalar `metalu-client.exe` en 2 PCs distintas en misma LAN
4. Doble-click cliente → descubre server automáticamente → login funciona
5. CRUD un cliente desde PC #1 → aparece en PC #2
6. Backup en `/admin/backups` → archivo `.pglitebackup` aparece
7. Restore con archivo válido → datos vuelven
8. Apagar server → clientes muestran "Conectando..." → encender → reconectan ≤30s
9. Cambiar IP server (DHCP) → cliente con cache vieja: matar y reabrir → re-discovery funciona

## Migration from old spec

| Old spec (2026-07-05) | New design |
|---|---|
| Server = Tauri shell + spawn Node child | Server = Node PKG bundle, no Tauri |
| `src-tauri/` with dual build configs | `src-tauri-client/` single client config |
| `BUILD_TARGET=server\|client` env var | Eliminated — server is separate product |
| Tauri Rust UDP discovery (server side) | Node `dgram.createSocket` announcer |
| Tauri Rust UDP discovery (client side) | Same — Rust listener (unchanged) |
| `tauri::generate_context!()` macro per-role | Once per binary (only client) |
| ~21GB Rust toolchain | Eliminated (server) — only client builds Rust |

## Open decisions for implementation plan

- **PKG vs nexe vs SEA**: PKG (`@yao-pkg/pkg`) chosen — most mature, handles WASM via `--public-files`. SEA (Node 21+) considered but requires Node 21+ runtime.
- **Cache TTL**: 30 days. Configurable via env var `METALU_CLIENT_CACHE_DAYS` (default 30).
- **Backup retention**: keep last 10 backups, auto-prune older.
- **Restore lock TTL**: 5 minutes max (force-release if stale).
- **UDP port**: 3001 fixed (configurable via env var `METALU_DISCOVERY_PORT`).
- **First-user bootstrap**: seed creates `admin/admin` on first run. Subsequent users via admin UI (existing flow).

## Risks

- **`pkg` is community-maintained** — official Node SEA is the future direction. Mitigation: document migration path in README.
- **PGlite WASM inside PKG bundle** — must extract to temp dir at runtime (PKG `--public-files` handles this).
- **WebView2 missing on Windows 10** — Tauri shows error on launch. Mitigation: detect + show modal with WebView2 installer link.
- **Single-writer ceiling** — PGlite serializes; 6+ concurrent writes/sec may timeout. Acceptable for 2-5 PCs LAN.
- **`app_data_dir` ownership** — if `%APPDATA%/metalu/` is cloud-synced (OneDrive), PGlite file conflicts. Documented warning in /admin.
- **No TLS** — LAN assumed trusted. If workshop has guests on WiFi, document WPA2 isolation requirement.
